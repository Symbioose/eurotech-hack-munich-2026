## 2. What is fully working
Features that run end-to-end on the live app, with real data and real logic. Be specific: name the feature, what input it takes, what output it produces.

- **AI pipeline (OpenAI GPT-4.1-mini when configured)** — user types a plain-English deployment problem; the app runs a typed context → compliance → component → assembly → BOM → DfMA → supplier → scene pipeline and streams results via SSE. With `OPENAI_API_KEY`, context/component/RFQ/scene steps use structured JSON calls. Without a key or on runtime failure, the app uses deterministic parser/rule/catalog fallbacks so the demo remains runnable.
- **World model (PyTorch backend)** — a 2-layer GRU + self-attention network is trained on synthetic BuildGuard degradation trajectories generated by our simulator. The simulator uses hard-coded HK climate normals and degradation formulas inspired by public climate/battery/corrosion/seal references; raw NASA/HK datasets are not loaded directly in this repo. The backend includes a persisted `world_model.pt`, CEM planner, FastAPI endpoints, and `evaluate.py`. Latest evaluation against the synthetic simulator: component MAE ≈ 0.014, failure-probability MAE ≈ 0.022, autoregressive drift ≈ 3x over 100 steps.
- **3D node visualization** — Three.js / React Three Fiber renders the smart-city node with explode mode, x-ray, per-component risk colour ramp (green → red), damage tooltips, occlusion check, orbit snap on component select, and a pause/resume rotation control.
- **MCP servers** — local Node.js stdio MCP servers exist for compliance, hardware, supplier routing, scene generation, and source research. Core tools read checked-in JSON data. Tavily-backed tools only return candidate web evidence when `TAVILY_API_KEY` is set; they do not automatically promote findings into trusted catalog/compliance data.
- **Marketplace redirect with click logging** — every "Buy" click goes through `/api/go`, is logged to `data/_marketplace-clicks.jsonl`, tagged with UTM params, and 302-redirected to the real distributor search page (LCSC / Digi-Key / Octopart).
- **Export** — PDF readiness pack (jsPDF, full A4 layout), machine-readable JSON, and CSV download all work client-side.
- **Context gate** — the pipeline refuses to run if the input is not a hardware deployment problem; it uses GPT-4.1-mini when `OPENAI_API_KEY` is configured and a deterministic local gate otherwise.

## 6. Known limitations & next steps
What we would build next, and the weak spots we already know about. Naming these honestly is a strength, not a flaw.

- **World model validation is simulator-only** — the backend/model/planner/evaluation are real, but the checkpoint is trained and evaluated against synthetic simulator trajectories, not deployed BuildGuard hardware logs; accuracy on real deployments is unknown.
- **Component catalog and compliance rules are static** — new components, price changes, EOL notices, and regulation updates require a manual file edit; a live Octopart/Nexar + HK government feed integration would fix this.
- **Single city (Hong Kong / GBA)** — deployment context is filtered to HK; extending to other smart-city contexts (Singapore, Dubai, Munich) requires adding compliance and climate datasets for those cities.
- **No user auth or server-side project persistence** — projects are stored in `localStorage`; a backend database would be needed for team sharing and history.
- **No real supplier verification** — the GBA supplier route is illustrative; real procurement would require verified contacts, NDA flows, and MOQ negotiation.
- **Scene MCP is not yet mandatory in every path** — deterministic generation uses required Scene MCP; the OpenAI scene path validates and fills scene nodes from catalog metadata instead of always calling Scene MCP.
